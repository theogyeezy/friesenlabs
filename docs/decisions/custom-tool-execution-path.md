# Custom-tool execution path: EnvironmentWorker vs client-side custom tools

**Decision needed before live wiring.** TODO.md:312 names this exact open question: the repo declares every tool as an MA *custom* tool on the agents while simultaneously building a worker to execute those same tools off the environment work queue. Per the current Managed Agents docs, those are two different mechanisms that never meet — and one of them, as authored, cannot work.

## Context (what the code does today)

**Every Uplift tool is declared to MA as a client-side custom tool.** `Tool.to_spec()` (`agents/tools/base.py:66-73`) serializes each registry tool as `{"type": "custom", name, description, input_schema}`. `ManagedAgentsRuntime._tool_specs` (`agents/runtime.py:97-105`) puts those specs on every agent at `create_agent` time, prepending the built-in `{"type": "agent_toolset_20260401"}` (`runtime.py:26,103`). The registry (`agents/tools/registry.py:17-21`) holds 9 tools: 5 read-only AUTO (search_rag, query_cube, read_crm, run_model, build_view), 1 draft AUTO (draft_email), 3 side-effecting ALWAYS_ASK (send_email, update_deal, issue_quote).

**The same tools are also registered with an EnvironmentWorker.** `worker/worker.py:33` builds `TOOLS = [SearchRag(), QueryCube(), ReadCrm(), DraftEmail(), SendEmail(), UpdateDeal(), IssueQuote()]` and `run()` (`worker.py:127-161`) passes them to `EnvironmentWorker(client, environment_id=, environment_key=, workdir=, tools=_tools_for_poll, context_factory=_context_for)`. The `tools=` and `context_factory=` kwargs are self-flagged `# VERIFY` (`worker.py:157-160`) — and per the current SDK docs they are **not documented EnvironmentWorker surface** (see Sources).

**The conversation layer assumes the worker exists.** `conv/session.py:214-264` (`_handle_coordinator`) routes `agent.custom_tool_use` events surfaced by `send_message` (`agents/runtime.py:265-276`): side-effecting names resolve through the trusted registry and `invoke()` → Greenlight proposal (draft-only holds); read-only events are passed through untouched on the explicit theory that "the self-hosted worker executes read tools in the VPC" (`session.py:23-26, 238`).

**Nobody ever closes the loop.** `grep custom_tool_result` across the repo returns zero hits — no code path ever sends `user.custom_tool_result` back to the session. `send_message` breaks on every `session.status_idle` including `requires_action` (`runtime.py:284-299`). So on today's code, **any coordinator tool call — read or write — leaves the MA session blocked in `requires_action` forever**, on either path. This is not a nit; it's the live-wiring bug the decision unblocks.

**The worker has real infra cost and real blockers.** `infra/modules/worker/main.tf`: a 24/7 Fargate service (desired_count 1, 0.5 vCPU / 1 GB ARM64 + ADOT sidecar) ≈ **$14–15/mo** — explicitly PARKED as a cost call because run-rate is ~$185/mo against the $200 budget alarm (TODO.md:159, BUILD_STATUS 2026-06-09). It is also blocked on the **Console-only environment key** (`sk-ant-oat01-…`, no SDK mint — TODO.md:456-458) with no rotation runbook (TODO.md open question). Meanwhile `api/asgi.py:131+` (`make_executor`) already gives the API task a complete registry-dispatch executor with tenant-bound DB/RAG/Cortex/Greenlight clients — the API task already has everything needed to execute tools in-process.

**One more wrong shape:** `Greenlight.to_ma_confirmation` (`api/control/greenlight.py:192-198`) emits `user.tool_confirmation` with an `edited_input` field. Per the docs, `user.tool_confirmation` is the reply for **server-executed** tools under `always_ask` permission policies (agent toolset / MCP) — custom tools explicitly have no permission policy ("you're the one executing") — and its documented fields are `tool_use_id`, `result: allow|deny`, `deny_message`; `edited_input` is not documented. For an approved custom-tool action, the correct reply is a `user.custom_tool_result` after your side executes.

### What the MA docs actually say (verified via the claude-api skill, June 2026)

1. **Custom tools are client-side by definition.** Agent emits `agent.custom_tool_use` on the session event stream → session idles (`stop_reason: requires_action`) → *your application* executes → you send `user.custom_tool_result` (with `custom_tool_use_id`) via `sessions.events.send`. The SDK README ships the exact stream → collect calls → break on idle → execute → send results → re-stream loop. This is also the documented pattern (client-patterns "Pattern 9") for keeping credentials host-side — the container/worker never sees them.
2. **The self-hosted EnvironmentWorker serves the *built-in agent toolset*** (bash, read, write, edit, glob, grep) for `config: {"type": "self_hosted"}` environments. Documented constructor: `EnvironmentWorker(client, environment_id=, environment_key=, workdir=)`; tool customization goes through `AgentToolContext` + `beta_agent_toolset(env)` + the lower-level `tool_runner()` — there is **no documented mechanism that routes `type: "custom"` tool calls to the environment work queue.** Even in a self_hosted environment, the docs' tools table says custom tools are executed by "your application" via the event round-trip.
3. **Permission policies don't apply to custom tools.** `always_ask` → `user.tool_confirmation` exists only for agent-toolset/MCP tools. Greenlight must live in whatever executes the custom tool — which the Phase 4 `Tool.invoke` base class already guarantees (`agents/tools/base.py:80-102`).

Bottom line: the worker, as authored, registers custom tools with a class that will never be handed them. The conflation isn't a style problem — one of the two paths is a documented dead end for Uplift's tool surface.

## Options

### Option 1 — Client-side custom tools: the API orchestrator executes everything (no worker)

Rework `send_message`/`_handle_coordinator` into the documented loop: drain stream → on `agent.custom_tool_use`, execute AUTO tools in-process through the **same** `api/asgi.make_executor` registry seam (tenant-bound RLS clients already wired), send `user.custom_tool_result`, keep draining; for ALWAYS_ASK tools, route to Greenlight and immediately reply with a `custom_tool_result` saying "queued for human approval, not performed" (exactly the pattern `runtime_selfhosted.py:331-339` already uses), then dispatch the real action through the existing post-approval gate when a human approves. Drop the implied `agent_toolset_20260401` from `_tool_specs` (no worker will serve bash calls; a stray built-in tool call would hang the queue forever). Fix/retire `to_ma_confirmation`.

- **Cost:** $0 new infra. Kills two of the four parked owner blockers (Console env-key click; worker deploy cost call). No env-key custody/rotation problem at all.
- **Latency:** one SSE hop; tools are millisecond Aurora/pgvector queries running in the task that already binds `app.current_tenant`. Strictly better than queue → long-poll → claim → execute → post-back.
- **Greenlight:** first-class. The orchestrator is the gate; draft-only is enforced by the same base class on the same trusted registry as the HIPAA fallback — `ManagedAgentsRuntime` and `SelfHostedToolUseRuntime` become structurally symmetrical (one mental model behind the `runtime.py` seam).
- **Risk / effort:** must implement reconnect-with-consolidation (SSE drop while a `custom_tool_use` is pending is a documented deadlock; the fix — `events.list` + dedupe-by-id on reconnect — is documented). Tool execution load lands on API tasks (fine at current scale; it's the same work `make_executor` already does for approved actions). Effort: ~days, and the SDK README contains the reference loop.

### Option 2 — EnvironmentWorker: deploy the Fargate worker and serve tools off the queue (as authored)

- **Cost:** +$14–15/mo against a budget already at ~$185/$200, plus the env-key Console click, plus a rotation runbook that doesn't exist.
- **Risk:** built on undocumented SDK surface (`tools=`, `context_factory=` kwargs); per the docs custom-tool calls never reach the work queue, so the worker would sit polling for built-in-toolset work Uplift's agents shouldn't be doing; tenant binding via "session metadata at the worker" is a second unverified seam; the `workers_polling`-per-poll heartbeat assumption is a third (TODO.md:314). Greenlight is awkward here even in the best case — the worker must return *something* synchronously while an approval sits pending for hours.
- **Effort:** high, with a strong chance the live verification simply proves the docs right and the path dead.

### Option 3 — Hybrid: Option 1 now, worker IaC stays shelved for a real sandbox need

Same as Option 1, plus an explicit decision record: keep `infra/modules/worker` authored-but-gated (`worker_deployed=false`, `infra/variables.tf:128-131`) for the day an Uplift feature genuinely needs the built-in sandbox toolset in-VPC (e.g., code-exec analytics over tenant files). Rewrite `worker/worker.py` to drop its claim to serve registry custom tools; if/when revived it serves the agent toolset only, via the documented `beta_agent_toolset` path. Cost of keeping the module: zero (it's not applied).

## Recommendation

**Option 3 — which is Option 1 operationally: client-side custom tools win; the orchestrator (API task) executes all nine registry tools over the `agent.custom_tool_use` → `user.custom_tool_result` round-trip, with Greenlight inline; the EnvironmentWorker is removed from the live path and its IaC shelved.**

Rationale, in order of force: (1) it's the only path the documented API supports for `type: "custom"` tools — the worker would literally never be handed them; (2) it's $0/mo versus a parked cost call, and it deletes the Console-env-key blocker and rotation problem outright; (3) it puts Greenlight where it belongs — the entity executing the tool — preserving the draft-only base-class guarantee with zero new seams; (4) it makes the managed and HIPAA runtimes the same shape behind `runtime.py`, which is the whole point of the seam; (5) it fixes the real bug either way: today nothing sends `user.custom_tool_result`, so every tool-calling session deadlocks. The worker was designed for a problem Uplift doesn't have (in-VPC *sandbox* tools) using SDK surface that doesn't exist (custom tools via the work queue). Don't deploy it; don't keep pretending `conv/session.py`'s read-tools-run-in-the-worker comment is true.

Concrete next steps: (a) rework `send_message` into the execute-and-resume loop with reconnect-consolidation; (b) drop `AGENT_TOOLSET` from `_tool_specs` (or gate it on a future sandbox feature); (c) rewrite `worker/worker.py`'s docstring/role and keep the module unapplied; (d) replace `to_ma_confirmation` with the approve→execute→`user.custom_tool_result` flow; (e) update TODO.md:312/314 and the `session.py:23-26` routing comment.

## What would flip this

- **A real sandbox need.** If an Uplift agent must run bash/file-ops/code over tenant files inside the VPC (Agent Studio-style analysis), deploy the shelved worker *for the built-in toolset only* — custom tools stay client-side regardless. This is additive, not a reversal.
- **Anthropic ships custom-tool dispatch through environment work queues.** Re-check at the next MA beta revision; if `type: "custom"` calls can be served by the worker with per-call context, the latency/cost calculus gets re-run (the queue hop still probably loses for millisecond DB tools).
- **Live behavior contradicting the docs.** The repo's own VERIFY flags (worker.py:157-160, runtime.py:289-291) call for a live smoke; if a live self_hosted session demonstrably routes custom-tool work items to the queue, this brief's premise is wrong — revisit before deleting code.
- **API-task scaling pain.** If holding long SSE drains plus in-request tool execution ever destabilizes the API service, introduce an *Uplift-side* internal queue/executor — that's an architecture change on our side, not a reason to adopt the MA environment queue.

## ⚠️ CRITIC CORRECTIONS (read before ratifying)
**The recommendation above rests on refuted claims — treat it as REVISED below.**
- Claim 1 (load-bearing clause): 'there is no documented mechanism routing custom-tool calls to the self-hosted environment work queue' is wrong. The installed SDK (anthropic 0.109.0 in the repo venv; lockfile pins 0.109.1) ships exactly that mechanism: the EnvironmentWorker claims the session work item and drives a SessionToolRunner that 'dispatches every agent.tool_use AND agent.custom_tool_use event against a local tool registry, posts the matching result event back (user.tool_result / user.custom_tool_result)' (.venv/.../anthropic/lib/tools/_beta_session_runner.py:5-7), with an explicit split-client partial-fulfilment contract (lines ~308-320) where the worker answers the custom tools it owns and leaves unowned ids pending for the app backend. A self-hosted worker registering Uplift's registry tools by name would execute them and answer user.custom_tool_result — which is precisely the repo's design intent.
- Claim 2: 'the tools= ... kwargs ... are not documented SDK surface' is wrong for tools=. EnvironmentWorker.__init__ accepts tools: EnvironmentWorkerTools | None (._worker.py:259) and its docstring (lines 225-229) documents it as 'a fixed list, or a callable' receiving the per-session AgentToolContext. Only context_factory= is nonexistent (it does raise TypeError, as the heartbeat brief found).
- Claim 4's conclusion: sessions are blocked today because the worker is undeployed and would crash on the nonexistent context_factory kwarg — not because the worker path architecturally cannot answer custom tool calls. With that kwarg deleted and the repo Tool ABCs wrapped as SDK runnable tools (already tracked in TODO.md 'Worker tool adapter: wrap repo Tool ABCs as the SDK BetaFunctionTool type'), the SDK answers user.custom_tool_result for worker-owned tools. The 'EnvironmentWorker is solving a problem Uplift doesn't have' framing collapses; the choice is a cost/ops tradeoff between two viable executors, not a forced architecture.

**REVISED recommendation:** BOTH paths are SDK-supported (the critic verified EnvironmentWorker's SessionToolRunner dispatches `agent.custom_tool_use`, and `tools=` is a documented kwarg; only `context_factory=` is nonexistent — see the heartbeat brief). The real trade-off: worker path = a 24/7 Fargate service + the Tool→SDK adapter (already a TODO) but no orchestrator round-trip; client-side = no standing worker cost but the API task must drive tool execution over SSE. For v1 (pre-revenue, $200 budget): lean CLIENT-SIDE to avoid the standing worker, and adopt the worker path when session volume or tool latency justifies it. Ratify with that framing, not the original 'no documented mechanism' claim.
